# 消息队列？队列？什么是队列？

队列是一种特殊的线性表，特殊之处在于它只允许在表的前端（front）进行删除操作，而在表的后端（rear）进行插入操作，和栈一样，队列是一种操作受限制的线性表。进行插入操作的端称为队尾，进行删除操作的端称为队头。队列中没有元素时，称为空队列。

# 什么是消息队列

“消息队列”是在消息的传输过程中保存消息的容器。消息队列[^**中间件**]是[^**分布式系统**]中重要的组件。

# 为啥用消息队列？

当公司本身的业务体量很小时，直接使用单机技术栈都能搞定，由于业务体量不断扩大，采用[^**微服务的设计思想**]，[^**分布式的部署方式**]，拆分了很多的服务，随着体量的增加以及业务场景越来越复杂了，很多场景[^**单机的技术栈**]和[^**中间件**]已经不够用了，而且对系统的友好性也下降了，于是引入消息队列中间件。

# 什么场景用到了消息队列？解决什么问题

**消息队列的经典场景:**<font color=red>[^异步、削峰、解耦]</font>

主要解决应用解耦，异步消息，流量削峰等问题，实现高性能，高可用，可伸缩和最终一致性架构。

目前使用较多的消息队列有Kafka，RocketMQ，RabbitMQ，ZeroMQ，ActiveMQ，MetaMQ



- 应用解耦：让**不同服务专注自身业务**，通过消息队列交换信息，如营销系统和支付系统分开。
- 异步消息：例如电商订单系统中，下单后订单处理、库存扣减、支付处理等环节可**异步进行**，提高系统响应速度。
- 流量削峰：避免流量过大冲垮系统，例如电商大促期间，通过消息队列缓存订单请求，避免系统崩溃。



## 生产者消费者模型

生产者消费者模型是一种经典的并发编程设计模式，用于解决多个线程（或进程）之间的协作和数据传递问题，核心是通过一个“缓冲区”（也叫消息队列）实现生产者和消费者的解耦与协同。



- 生产者：负责生成数据（如任务、消息、数据对象等），并将数据放入缓冲区。

- 消费者：从缓冲区中取出数据并进行处理（如执行任务、解析消息等）。

- 缓冲区：连接生产者和消费者的中间容器（可以是队列、链表等结构），用于平衡两者的处理速度差异。



# Kafka（高吞吐量的分布式日志系统）

Kafka 是一个分布式的基于发布 / 订阅模式的消息队列（Message Queue），主要应用于大数据实时处理领域。



消息队列中每个消息称为一个节点，每个节点有个序号（offset）来记录消息的位置，消费者根据自己的能力能消费多少消费多少，不断更新已处理的offset值，来不及处理的消息会堆积在内存里，如果消费者更新重启，消息就丢了，可以将队列挪出来，变成一个单独的进程，即消息队列。



<img src="https://gitee.com/aijiu_4566/jacobs-ladder/raw/master/%E5%90%8E%E7%AB%AF/static/Kafka.png" style="zoom:50%;" />



**高性能**：扩展更多的消费者，同时可以增加更多的生产者，提高消息的吞吐量，但会造成对消息的争抢，可以对消息进行分类，每一类是一个topic，再将每个topic队列拆分成好几段，每段就是一个partition分区，每个消费者负责一个partition就大大降低了争抢，提升了消息队列的性能。



**高扩展性**：随着partition变多，如果partition都在同一台机器上的话，就会导致单机CPU和内存过载，影响整体性能，我们可以将partition分散部署在多台机器上，每一台机器就代表一个broker，我们可以通过增加broker来缓解机器CPU过载带来的性能问题。



**高可用**：如果其中一个partition所在的broker挂了，那broker里所有partition的消息都没了，可以给partition多加几个副本，统称为replicas将他们分为leader和follower，leader负责生成者和消费者的读写请求，而follower只管同步leader的消息，将leader和follower分散到不同的broker上，leader所在的follower挂了也不会影响到follower所在的broker，在follower中选举出新的（如何选举？）leader-partition顶上，这样就保证了消息队列的高可用性。



**持久化和过期策略**：假设所有broker都挂了岂不是数据全丢了，为解决这个问题，不能只把数据放在内存里，还要放在磁盘里（磁盘有限得加上保留策略即retention policy（磁盘数据超过一定大小或放置超过一定时间就会被清理掉））



**consumer group**：现在每次新增的消费者只能跟着最新的offset接着消费或从头开始，怎么让新增的消费者从某个offset开始消费呢？我们可以将消费者分组，不同消费者组维护自己的消费进度，互不干扰，新增的消费者按需加入组就好了。



**ZooKeeper**：组件太多，而且每个组件都有自己的数据和状态，所有还需要一个组件去统一维护这个组件的状态信息，即ZooKeeper（动物园管理员）组件，它会定期和broker通信，获取整个Kafka集群的状态，以此判断某些broker是不是挂了，某些消费组消费到哪了，但kafka从2.8.0版本支持移除ZooKeeper。



**实际案例：电商用户行为日志收集**

>
> append-only和不可变的特性带来显著的IO局部性优化：
>
> - 内存中可通过缓冲区（buffer）批量处理写入；
> - 操作系统层面利用页缓存（page cache）减少磁盘直接IO；
> - 磁盘上消息集中存储在连续区域（同一磁道），顺序写入效率远高于随机读写。
> - 这也是MySQL binlog、Redis AOF等采用类似设计的原因——最大化写入性能
>
> ------
>
>  它是典型的IO密集型应用 所以它并不是线程池，Kafka的大量技术细节都在解决IO性能，包括单不限于零拷贝
>
> Kafka的性能瓶颈主要在IO（磁盘读写、网络传输），而非CPU，因此其设计侧重优化IO而非依赖线程池并发：
>
> - 采用单线程处理单个分区的消息读写（避免多线程竞争）；
> - 通过零拷贝（zero-copy）等技术减少数据在用户态与内核态之间的拷贝开销，进一步降低IO成本。

## 架构上做减法：

**简化协调节点**：

ZooKeeper在kafka架构中会和broker通信维护kafka集群信息，一个新的broker连上ZooKeeper后，其他broker就能立马感知到它的加入，像这种能在分布式环境下让多个实例同时获取同一份信息的服务就是所谓的**分布式协调服务**



**ZooKeeper**：ZooKeeper作为一个通用的分布式协调服务它不仅可以用于服务注册和发现，还可以用于分布式锁配置管理等场景，kafka只用到了它的部分功能，而ZooKeeper就显得冗余了于是RocketMQ直接将ZooKeeper去掉换成了NameServer，用一种更轻量的方式管理消息队列的集群信息。开发kafka的人后来也意识到了ZooKeeper过重的问题所以从2.8.0版本就支持移除ZooKeeper通过broker之间加入一致性算法Raft实现同样的效果这就是所谓的kRaft或Quorum模式



**简化分区**：kafka会将topic拆分为多个partition用来提升并发性，在RocketMQ里也一样，将topic拆分成了多个分区但换了个名字叫Queue(队列)，kafka中的partition会存储完整的消息体，而RocketMQ的Queue却只存一些简要信息，比如消息偏移offset，而消息的完整数据则放到一个叫CommitLog的文件上通过offset我们可以定位到CommitLog上的某条消息。



**消费对比**：kafka消费消息，Broker只需要直接从Partition读取消息返回就够了，而在RocketMQ中Broker则需要先从Queue上读取到offset的值再跑到CommitLog上将完整数据读出来，也就是需要读两次。



**看起来kafka的设计更高效，为什么RocketMQ不采用kafka的设计？**



**底层存储**：**顺序写速度大于随机写**



**kafka**：kafka的Partition分区其实在底层有很多段也就是segment组成，每个segment可以认为就是个小文件将消息数据写入到Partition分区，本质上就是将数据写入到某个segment文件下，写磁盘，顺序写的性能会比随机写快很多，差距高达几十倍，为了提升性能，kafka对每个小文件都是顺序写，如果只有一个segment文件，那写文件的性能会很好但当topic变多后topic底下的Partition分区也会变多，对应Partition底下的segment文件也会变多，同时写多个topic底下的Partition就是同时写多个文件，虽然每个文件内部都是顺序写的，但多个文件存放在磁盘的不同地方，原本顺序写磁盘就可能劣化变成了随机写，于是写性能就降低了。



**RocketMQ**：为了缓解同时写多个文件带来的随机写问题RocketMQ索性将单个Broker底下的多个topic数据全部写到一个逻辑文件CommitLog上，这就消除了随机写多文件的问题将所有写操作都变成了顺序写，大大提升了RocketMQ在多topic场景下的写性能（怎么按分类查询的？）



**简化备份模型**：kafka会将Partition分散到多个Broker中并为Partition配置副本将Partition分为leader和follower也就是主和从主从Partition之间会建立数据同步本质上就是同步Partition底下的segment的文件数据，RocketMQ将Broker上的所有topic数据写到CommitLog上，所有RocketMQ索性直接同步CommitLog文件以Broker为单位区分主从保持高可用的同时，也大大简化了备份模型。



## 在功能上做加法：

**消息过滤**：kafka支持通过topic将数据进行分类比如订单数据和用户数据是两个不同的topic，如果还想再一部分类呢？比如同样是用户数据我们还要根据vip等级分类我们只需要vip6的数据，kafka中需要消费topic为用户数据的所有消息再将vip6的用户过滤出来，而RocketMQ支持对消息打上标记，也就是打tag，消费者能根据tag过滤（如何过滤的？）所需要的数据，这样消费者就能只获取这部分数据省下了消费者过滤数据时的资源消耗。



**支持事务**：kafka支持事务比如生成者发3条消息ABC，要么同时发送成功要么同时失败但更我们要的不太一样，写业务代码时我们更想要的事务是执行一些自定义逻辑和生产者发消息这两件事要么同时成功要么同时失败，而这正是RocketMQ支持的事务能力。



**延时队列**：如果我们希望消息被投递出去后，消费者不能立马消费到，而是过一定时间后才能消费，也就是所谓的延时消息，这可以用RocketMQ的延时队列实现，而kafka就得程序员自己实现类似功能。



**死信队列**：消费消息有可能失败，失败后一般可以设置重试，如果多次重试失败，RocketMQ会将消息放到一个专门的队列方便我们后面单独处理即死信队列，kafka原生不支持该功能。



**消息回溯**：kafka支持通过调整offset让消费者从某个地方开始消费，而RocketMQ除了可以调整offset还支持调整时间。

## 零拷贝是什么？

消息队列的消息为了防止进程崩溃后丢失，一般不会放在内存里而是放在磁盘上。操作系统分为用户空间和内核空间，程序处于用户空间，而磁盘属于硬件，程序需要操作系统去调用硬件能力，如果用户想要将数据从磁盘发送到网络那么就会发生下面这几件事程序、发起系统调用read()、将磁盘数据从设备拷贝到内核空间的缓冲区、再从内核空间的缓冲区拷贝到用户空间、程序再发起系统调用write()，将数据从用户空间拷贝到Socket发送缓冲区、再从socket发送缓冲区拷贝到网卡，最终数据就会经过网络到达消费者，**整个过程本机内发生了两次系统调用，对应四次用户空间和内核空间的切换以及四次数据拷贝，来回拷贝，为了优化，使用零拷贝技术**。



**零拷贝技术常见的有两种：分别为mmap和sendfile**



**mmap**：mmap()是操作系统内核提供的一个方法，可以将内核空间的缓冲区映射到用户空间，用了它整个发送流程就有了些变化，程序发起系统调用mmap()将磁盘数据从设备拷贝到内核空间缓冲区，再映射到用户空间不需要拷贝，程序再发起系统调用write()数据从内核空间缓冲区拷贝到Socket的发送缓冲区，在拷贝到网卡，**整个过程发生两次系统调用，对应四次用户空间和内核空间的切换以及三次数据拷贝，省下一次内核空间到用户空间的拷贝，0内核空间到用户空间拷贝**。



**sendfile**：sendfile()也是内核提供的一个方法，用来发送文件数据，程序发起系统调用sendfile()将磁盘数据从设备拷贝到内核空间缓冲区，然后直接拷贝到网卡，**整个过程发生一次系统调用，对应2次用户空间和内核空间的切换以及2次数据拷贝，0CPU拷贝，不需要CPU参与而是DMA控制器在干活不耽误CPU跑程序**。



RocketMQ使用的是mmap技术，而kafka使用的是sendfile技术，kafka以更少的拷贝次数以及系统内核切换次数获得了更高的性能

所有为啥RocketMQ不适应sendfile技术


**mmap返回的是数据的具体内容应用层能获取到消息内容并进行一些逻辑处理而sendfile返回的则是成功发送了几个字节数具体发了什么内容应用层根本不知道，而RocketMQ的一些功能却需要了解具体的消息内容方便二次投递等**比如将消费失败的消息重新投递到死信队列中，如果使用sendfile根本没机会获取消息内容，也就没办法实现一些好用的功能了。而kafka就没有这些功能特性，追求极致性能正好可以使用sendfile，除此之外，kafka高性能的原因还有很多比如批处理数据压缩但这些RocketMQ都能借鉴唯独零拷贝无法借鉴。



# RabbitMQ（企业级消息代理）

**场景**：A服务负责转发用户请求到B服务，B服务是个算法服务，GPU资源有限，当请求量大到B服务处理不过来的时候，希望能够优先处理会员用户的请求，那么如果普通用户和会员用户同时发送请求怎么办？



**Queue**：为了更好的管理不同种类的数据，提供多个队列，生产者可以自定义Queue的名字，并根据需要将消息投递到不同的Queue中，每个Queue都设计为独立的进程，某个进程挂了不影响其他进程的正常工作。



**Exchange**：有些生产者把消息发到一个Queue中，而有些则发送给多个Queue，甚至广播给所有Queue，于是我们还需要一个可以定制消息路由分发策略的组件，交换器Exchange，将它与Queue绑定在一起，通过一个类似正则表达式的字符串binding Key声明绑定的关系，让用户根据需要选择要投递的队列，这些维护在Exchange里的路由方式和绑定关系，我们称为元数据。



**RabbitMQ**：包含多个Queue进程和Exchange组件的消息队列就RabbitMQ，每个服务器上的RabbitMQ实例就代表着一个Broker



![image-20250803194833800](https://gitee.com/aijiu_4566/jacobs-ladder/raw/master/%E5%90%8E%E7%AB%AF/static/RabbitMQ.png)



**功能特性**：延时队列，死信队列，优先级队列



**优先级队列**：在生成者发送消息时，为消息标记上优先级，消费者总是消费优先级高的消息，当高优消息处理完后再处理普通消息。

**普通集群模式**：每个Broker都是一个完整功能的RabbitMQ实例，都能进行读写，它们之间会互相同步Exchange里的元数据，但不会同步Queue数据。

假设Q1，Q2，Q3分别部署在Broker1,2,3中

**对于写操作**，生成者将消息写入Broker1中的Queue1后，并不会同步给其他Broker，但如果此时Broker1中的元数据发生变化，则会将元数据同步到其他两个Broker中。

**对应读操作**，消费者想要读取Queue1数据时，如果访问的是Broker1，则直接返回Queue1中的数据，如果访问的是Broker2，Broker2则会根据Exchange中的元数据从broker1那读取数据在返回给消费者，这样就可以通过增加Broker提升RabbitMQ集群整体吞吐量，保证了扩展性。

虽然支持读写Queue的数量是增加了但对于单个Queue本身的读写能力并没有提升，而且更重要的是，每个broker依然有单点问题，broker之间并不同步Queue里面的数据，某个Queue所在的数据要是挂了，就没法读写这个Queue了，**缺少高可用性**。



------



**镜像队列集群**：将Queue在其他Broker中加几个副本，他们有主从关系，主Queue负责读写数据，从Queue（镜像队列）负责同步负责主Queue数据，一旦主Queue所在的Broker挂了，从Queue就可以顶上，成为新的主Queue**实现高可用**，

**对于写操作**，数据写入主Queue后，会将Exchange好Queue数据同步给其他Broker上。

**对于读操作**，消费者读取数据时，如果访问的是主Queue所在的Broker则直接返回数据，否则当前Broker会从主Queue所在的Broker上读取数据之后返回给消费者，**牺牲了吞吐量换取高可用**。



------



**Quorum队列集群**：怎么在多个节点之间同步数据？RabbitMQ基于erlang进行开发，erlang是个很特别的语言，它自带虚拟机和分布式通信框架，RabbitMQ通过分布式通信框架在broker间同步元数据，但如果broker间通信断开，镜像队列可能出现多个节点都认为自己是主节点的情况，导致数据不一致（脑裂问题），我们可以使用更靠谱的一致性算法raft来同步多个Broker的元数据和队列数据，通过引入选举机制来解决网络分区问题，这就是所谓的Quorum队列集群。

# 三种消息队列的对比

| 特性     | Kafka              | RabbitMQ           | RocketMQ           |
| -------- | ------------------ | ------------------ | ------------------ |
| 吞吐量   | 极高（百万级 TPS） | 中等（万级 TPS）   | 高（十万级 TPS）   |
| 延迟     | 高（毫秒级）       | 低（微秒级）       | 中（毫秒级）       |
| 消息顺序 | 分区内有序         | 队列有序           | 队列有序           |
| 事务支持 | 有限（需外部协调） | 支持（插件扩展）   | 原生支持           |
| 典型场景 | 日志收集、流处理   | 企业应用、复杂路由 | 电商交易、金融业务 |
| 代表用户 | LinkedIn、Netflix  | 携程、Airbnb       | 阿里巴巴、蚂蚁金服 |



**选择建议：**

**大数据场景**：选择 Kafka，如日志收集、实时数仓。
**复杂路由需求**：选择 RabbitMQ，如企业 ERP 系统（企业资源管理系统）。
**金融级事务**：选择 RocketMQ，如支付、订单系统。

# 总结

- Kafka 通过分区与顺序 I/O 实现高吞吐；
- RocketMQ 通过事务消息与主从同步保障金融级可靠性。
- RabbitMQ 通过灵活的 Exchange 路由满足复杂业务逻辑；
